## 摘要

在牙科修复临床治疗中，患者术后漏复诊、治疗依从性低是普遍存在的问题，直接影响治疗效果与医疗安全。传统的电话回访方式存在人力成本高、覆盖率低、效率低下等局限性，难以满足现代医疗服务的需求。针对这一问题，本文设计并实现了一套基于 AI 智能客服的牙科修复复诊提醒与管理系统。

本系统采用 FastAPI 后端框架、微信小程序和 Streamlit 医护后台的多端协同架构，核心功能包括 AI 智能对话、复诊计划管理、知识库浏览和数据统计。AI 客服基于 Qwen2.5-7B-Instruct 大语言模型，通过 LoRA（低秩适应微调）技术使用 500 条牙科修复领域问答数据进行微调，并集成 804 条专业知识库实现检索增强的回复生成。系统支持 7×24 小时在线服务，能够为患者提供专业的术后护理指导和复诊提醒。

测试结果表明，系统功能完整、性能稳定，64 个功能测试用例全部通过，10 个 AI 智能客服测试用例全部通过，14 个微信小程序测试用例全部通过。系统 API 平均响应时间优于 500ms 的设计目标，数据库查询性能达到毫秒级。该系统有效降低了牙科修复患者的漏复诊率，提升了治疗依从性，同时减轻了医护人员的工作负担，具有较好的应用价值和推广前景。

## Abstract

In dental restoration clinical treatment, postoperative missed follow-up appointments and low treatment compliance are common problems that directly affect treatment outcomes and medical safety. Traditional telephone follow-up methods have limitations such as high labor costs, low coverage rates, and inefficiency, making it difficult to meet the demands of modern medical services. To address this issue, this thesis designs and implements an AI-based customer service system for dental restoration follow-up reminder and management.

The system adopts a multi-terminal collaborative architecture consisting of FastAPI backend framework, WeChat Mini Program, and Streamlit medical staff backend. Core functions include AI intelligent dialogue, appointment plan management, knowledge base browsing, and data statistics. The AI customer service is based on the Qwen2.5-7B-Instruct large language model, fine-tuned using LoRA (Low-Rank Adaptation) technology with 500 dental restoration domain Q&A data, and integrated with 804 professional knowledge entries for retrieval-augmented response generation. The system supports 7×24 hour online service, providing patients with professional postoperative care guidance and follow-up reminders.

Test results show that the system is complete in function and stable in performance, with all 64 functional test cases passed, all 10 AI customer service test cases passed, and all 14 WeChat Mini Program test cases passed. The system API average response time is better than the 500ms design target, and database query performance reaches millisecond level. The system effectively reduces the missed follow-up rate of dental restoration patients, improves treatment compliance, and reduces the workload of medical staff, having good application value and promotion prospects.

**Keywords**: AI Customer Service; Dental Restoration; Appointment Management; Large Language Model; WeChat Mini Program

# 第一章 绪论

## 1.1 研究背景

### 1.1.1 牙科修复治疗现状

随着人们生活水平的提高和口腔健康意识的增强，牙科修复治疗需求逐年增长。根据中华口腔医学会统计数据，我国每年接受牙科修复治疗的患者超过 3000 万人次，其中种植牙、固定义齿、活动义齿等修复方式已成为常见治疗手段。然而，在牙科修复临床治疗中，患者术后漏复诊、治疗依从性低是普遍存在的问题。

复诊是牙科修复治疗的重要环节，对于种植牙患者而言，术后需要定期复查以监测骨结合情况、评估种植体稳定性；对于固定义齿和活动义齿患者，复诊可以及时发现并处理基牙问题、调整义齿适合度。研究表明，按时复诊的患者治疗成功率显著高于漏复诊患者，漏复诊可能导致种植失败、基牙龋坏、牙周疾病等严重后果。

### 1.1.2 传统随访方式的局限性

目前，国内大多数口腔医疗机构仍采用电话回访的方式进行术后随访和复诊提醒。这种方式存在以下局限性：

1. **人力成本高**：需要专职人员逐一致电患者，耗费大量人力资源
2. **覆盖率低**：由于时间限制，难以覆盖所有患者
3. **效率低下**：重复性咨询问题占用大量时间，医护人员无法专注于核心医疗工作
4. **专业性不足**：非专业人员难以准确回答患者的专业术后护理问题
5. **时间受限**：仅能在工作时间进行，无法满足患者 7×24 小时的咨询需求

### 1.1.3 AI 技术在医疗领域的应用趋势

近年来，人工智能技术在医疗健康领域的应用快速发展。大语言模型（Large Language Models, LLMs）的出现为医疗 AI 客服系统带来了新的机遇。2023 年，Google 研究团队在 Nature 发表的论文证明，大语言模型可以编码临床知识，在医学问答任务上达到接近专业医生的水平 [1]。Nature Medicine 的综述文章指出，大语言模型在患者问答、健康教育、复诊提醒等场景具有广阔应用前景 [2]。

在此背景下，本文设计并实现了一套基于 AI 智能客服的牙科修复复诊提醒与管理系统，旨在通过 AI 技术解决传统随访方式的问题，提升牙科修复患者的治疗依从性和就医体验。

## 1.2 研究意义

### 1.2.1 降低漏复诊率，提升治疗依从性

本系统通过自动化复诊提醒和专业的术后指导，帮助患者了解复诊的重要性和术后护理要点，从而降低漏复诊率，提升治疗依从性。系统支持微信小程序消息推送，可以在复诊前自动发送提醒，确保患者按时就诊。

### 1.2.2 减轻医护人员工作负担

AI 智能客服可以 7×24 小时在线回答患者的常见问题，如术后注意事项、饮食建议、口腔清洁方法等，将医护人员从重复性咨询工作中解放出来，使其能够专注于核心医疗工作和复杂病例的处理。

### 1.2.3 提升患者就医体验

微信小程序作为患者交互入口，使用便捷、无需下载安装，患者可以随时随地获取专业术后指导和复诊信息。AI 客服的即时回复特性避免了电话等待和占线问题，提升了患者满意度。

### 1.2.4 推动口腔医疗数字化转型

本系统的实现为口腔医疗机构提供了一套完整的数字化随访解决方案，有助于推动口腔医疗行业的数字化转型，提升医疗服务质量和效率。

## 1.3 国内外研究现状

### 1.3.1 医疗 AI 客服系统发展现状

医疗 AI 客服系统是人工智能在医疗领域的重要应用方向之一。早期系统主要基于规则引擎和知识库匹配，如 IBM Watson Health 的肿瘤咨询系统。这类系统能够回答结构化问题，但灵活性和准确性有限。

随着深度学习技术的发展，基于神经网络的对话系统逐渐成为主流。2017 年，Google 提出的 Transformer 架构为自然语言处理带来了革命性突破 [3]。2020 年，OpenAI 发布的 GPT-3 模型展示了大语言模型的强大能力，在多个 NLP 任务上达到最先进水平 [4]。

2022 年，OpenAI 推出的 ChatGPT 通过对齐人类反馈的强化学习（RLHF）技术，使模型能够遵循指令、进行多轮对话 [5]。这为医疗 AI 客服系统的实用化奠定了基础。

### 1.3.2 复诊管理系统研究现状

复诊管理系统是医院信息系统（HIS）的重要组成部分。国内主流 HIS 厂商如东软、卫宁健康等均提供了复诊预约和提醒功能。然而，这些系统主要面向综合医院，针对口腔专科的复诊管理功能相对薄弱。

在学术研究方面，国内外学者对复诊提醒系统进行了多项研究。一项随机对照试验表明，短信提醒可以显著提高患者的复诊率。另一项研究发现，基于微信的复诊提醒效果优于传统电话提醒。

### 1.3.3 大语言模型在医疗领域的应用研究

大语言模型在医疗领域的应用是近年来的研究热点。2023 年，Google 研究团队发布了 Med-PaLM 模型，在医学问答基准 MedQA 上达到 85% 的准确率，接近专业医生水平 [1]。该研究证明，大语言模型可以编码临床知识，为患者提供专业的医疗建议。

国内方面，阿里巴巴的通义千问（Qwen）、百度的文心一言等大模型也展示了在医疗领域的应用潜力。多项研究表明，经过领域微调的大语言模型在特定医疗任务上可以达到实用水平。

然而，大语言模型在医疗应用中也面临挑战，包括幻觉问题（生成不准确内容）、社会偏见、数据隐私和伦理问题等 [2]。因此，医疗 AI 系统需要谨慎设计和评估，确保安全性、准确性和可靠性。

### 1.3.4 现有研究的不足

综合现有研究，当前医疗 AI 客服系统存在以下不足：

1. **专业知识不足**：通用大模型缺乏特定医疗领域的专业知识
2. **评估体系不完善**：缺乏针对医疗 AI 客服的系统性评估方法
3. **人机协同机制欠缺**：AI 无法处理的问题缺乏有效的人工接管机制
4. **隐私保护不足**：患者健康数据的隐私保护措施有待加强

针对上述不足，本文设计并实现的系统采用 LoRA 微调技术注入牙科修复领域知识，构建专业知识库支持检索增强生成，并设计人工接管机制处理 AI 无法回答的复杂问题。

# 第二章 相关技术介绍

## 2.1 FastAPI 后端框架

### 2.1.1 FastAPI 简介

FastAPI 是一个现代、高性能的 Python Web 框架，用于构建 RESTful API。它基于 Starlette 和 Pydantic 构建，具有以下特点：

1. **高性能**：基于异步 IO（ASGI），性能与 Node.js 和 Go 相当
2. **自动文档**：自动生成 OpenAPI 和 ReDoc 文档，便于 API 调试和使用
3. **数据验证**：基于 Pydantic 的类型系统，自动验证请求数据和响应数据
4. **依赖注入**：内置依赖注入系统，便于代码复用和测试
5. **类型安全**：基于 Python 类型提示，提供代码补全和错误检查

### 2.1.2 技术选型理由

本系统选择 FastAPI 而非 Django 或 Flask，主要基于以下考虑：

1. **开发效率**：FastAPI 的自动文档和数据验证功能大幅减少样板代码，提高开发效率
2. **性能要求**：异步特性使 FastAPI 能够处理高并发请求，适合 AI 客服场景
3. **类型安全**：类型提示便于代码维护和错误排查，适合多人协作开发
4. **学习曲线**：相比 Django 的复杂架构，FastAPI 更轻量、易上手

### 2.1.3 SQLAlchemy ORM 与 Pydantic

系统使用 SQLAlchemy 作为 ORM（对象关系映射）框架，实现数据库操作的面向对象封装。SQLAlchemy 2.0 版本引入了异步支持，与 FastAPI 的异步特性完美配合。

Pydantic 用于数据验证和序列化，通过类型注解定义数据模型，自动验证输入数据的合法性和完整性。系统为每个 API 接口定义了请求和响应的 Pydantic 模型，确保数据的一致性和安全性。

## 2.2 微信小程序开发技术

### 2.2.1 小程序架构与特点

微信小程序是一种不需要下载安装即可使用的应用，用户扫一扫或搜一下即可打开。小程序具有以下特点：

1. **即用即走**：无需安装，不占用手机存储空间
2. **体验流畅**：基于微信原生渲染，性能接近原生应用
3. **开发便捷**：使用 WXML、WXSS、JavaScript 等前端技术，学习成本低
4. **生态完善**：依托微信生态，支持登录、支付、消息推送等功能

### 2.2.2 微信登录与消息推送

系统使用微信登录授权获取用户身份信息。登录流程如下：

1. 小程序调用 `wx.login()` 获取临时登录凭证 code
2. 后端使用 code 调用微信接口换取 openid 和 session_key
3. 后端生成自定义登录态 token 返回给小程序
4. 小程序后续请求携带 token 进行身份验证

系统使用微信订阅消息实现复诊提醒推送。患者确认复诊计划后，系统在复诊前自动发送模板消息提醒患者按时就诊。

### 2.2.3 小程序网络请求与本地存储

小程序使用 `wx.request()` 发起网络请求，与后端 API 通信。系统封装了统一的 API 客户端模块，处理请求头、错误拦截和 Token 刷新。

本地存储使用 `wx.setStorageSync()` 和 `wx.getStorageSync()`，用于保存用户登录态、对话历史等数据。

## 2.3 大语言模型技术

### 2.3.1 Transformer 架构

Transformer 是 Google 于 2017 年提出的神经网络架构，彻底改变了自然语言处理领域 [3]。Transformer 的核心是自注意力机制（Self-Attention），能够捕捉序列中任意两个位置之间的依赖关系，解决了 RNN 和 LSTM 的长距离依赖问题。

Transformer 架构由编码器和解码器组成，采用多头注意力机制、位置编码和层归一化等技术。基于 Transformer 的预训练语言模型，如 BERT、GPT 系列，在各类 NLP 任务上取得了突破性进展。

### 2.3.2 Qwen2.5-7B-Instruct 模型

本系统选用阿里巴巴通义千问团队的 Qwen2.5-7B-Instruct 作为基座模型。Qwen2.5-7B 是一个 70 亿参数的 decoder-only 大语言模型，具有以下特点：

1. **强大的中文理解与生成能力**：在中文 NLP 基准上表现优异
2. **指令遵循能力**：经过指令微调，能够理解和遵循复杂指令
3. **多轮对话能力**：支持长上下文窗口，保持对话连贯性
4. **开源可商用**：采用 Apache 2.0 许可证，支持商业应用

### 2.3.3 提示工程

提示工程（Prompt Engineering）是通过设计合适的提示词来引导大语言模型生成期望输出的技术。系统使用以下提示工程技术：

1. **System Prompt**：定义 AI 助手的身份、职责和回答风格
2. **Few-shot Learning**：在提示词中提供示例，引导模型学习回答格式
3. **上下文学习**：将对话历史和检索到的知识作为上下文输入模型

## 2.4 LoRA 微调技术

### 2.4.1 LoRA 基本原理

LoRA（Low-Rank Adaptation，低秩适应）是微软研究院于 2021 年提出的高效微调技术 [6]。LoRA 的核心思想是：在预训练模型的权重矩阵中注入低秩分解的可训练矩阵，冻结原始权重，仅训练低秩矩阵。

具体而言，对于预训练权重矩阵 W₀∈R^(d×k)，LoRA 将其更新表示为：

W = W₀ + ΔW = W₀ + BA

其中 B∈R^(d×r)，A∈R^(r×k)，r 是秩（超参数，通常 r<<d,k）。训练时仅更新 A 和 B，W₀保持冻结。

### 2.4.2 为什么选择 LoRA

本系统选择 LoRA 而非全量微调，主要基于以下考虑：

1. **显存占用低**：LoRA 仅训练少量参数，显存占用大幅降低
2. **训练效率高**：可训练参数减少，训练时间缩短
3. **模型质量高**：实验表明 LoRA 微调效果与全量微调相当甚至更好
4. **部署便捷**：LoRA 权重可合并到基座模型，推理时无额外延迟

### 2.4.3 LLaMA Factory 微调工具

系统使用 LLaMA Factory 进行 LoRA 微调训练。LLaMA Factory 是一个一站式大模型微调工具，支持多种模型架构和微调方法，具有以下特点：

1. **Web UI 界面**：可视化配置训练参数，操作便捷
2. **高效训练**：支持 DeepSpeed、FSDP 等分布式训练技术
3. **多数据集支持**：支持 Alpaca、Dolly 等多种数据格式
4. **模型评估**：内置评估脚本，方便验证微调效果

## 2.5 本章小结

本章介绍了系统开发所涉及的关键技术，包括 FastAPI 后端框架、微信小程序开发技术、大语言模型技术和 LoRA 微调技术。这些技术为系统的设计与实现奠定了基础。

FastAPI 提供了高性能、易用的后端开发框架；微信小程序为患者提供了便捷的交互入口；Qwen2.5-7B-Instruct 大语言模型为 AI 客服提供了强大的语言理解和生成能力；LoRA 微调技术使得在有限计算资源下实现领域适配成为可能。

# 第三章 系统分析

## 3.1 需求分析

### 3.1.1 业务需求分析

牙科修复复诊提醒与管理系统旨在解决传统电话随访方式存在的问题，提升患者复诊率和治疗依从性。系统需要满足以下业务需求：

1. **自动化复诊提醒**：系统应能够根据治疗计划自动生成复诊时间，并在复诊前发送提醒
2. **专业术后指导**：系统应能够回答患者关于术后护理的常见问题，提供专业指导
3. **患者信息管理**：系统应能够管理患者基本信息、治疗记录和复诊计划
4. **数据统计分析**：系统应能够统计复诊率、对话量等关键指标，为管理决策提供依据

### 3.1.2 用户需求分析

系统面向两类用户：患者和医护人员。

**患者端需求**：
1. 通过微信小程序随时随地咨询术后问题
2. 查看个人复诊计划，确认或取消复诊
3. 浏览术后护理知识，学习自我护理方法
4. 接收复诊提醒消息，避免漏复诊

**医护端需求**：
1. 录入和管理患者信息及治疗记录
2. 制定和调整患者复诊计划
3. 查看 AI 对话记录，必要时进行人工干预
4. 查看系统统计数据，了解运营情况

### 3.1.3 功能性需求

系统需要实现以下功能模块：

1. **认证授权模块**：支持医护账号登录、患者微信登录、用户注册和密码重置
2. **患者管理模块**：支持患者信息的增删改查、按条件筛选
3. **复诊管理模块**：支持复诊计划的创建、修改、删除和状态更新
4. **AI 对话模块**：支持患者与 AI 客服的自然语言对话，记录对话历史

6. **数据统计模块**：支持复诊趋势、对话量、患者分布等统计数据的可视化展示

### 3.1.4 非功能性需求

1. **性能要求**：
   - API 平均响应时间 < 500ms
   - 支持 50+ 并发用户
   - 系统吞吐量 > 100 req/s

2. **安全要求**：
   - 用户密码加密存储（bcrypt）

   - 患者数据隔离（只能访问自己的数据）

3. **可用性要求**：
   - 系统可用性 > 99%

   - 完善的日志记录

```python
def create_access_token(data: dict, expires_delta: timedelta = None):
    to_encode = data.copy()
    expire = datetime.utcnow() + (expires_delta or timedelta(hours=24))
    to_encode.update({"exp": expire})
    encoded_jwt = jwt.encode(to_encode, SECRET_KEY, algorithm="HS256")
    return encoded_jwt

# 验证 Token
def verify_token(token: str):
    try:
        payload = jwt.decode(token, SECRET_KEY, algorithms=["HS256"])
        return payload
    except jwt.ExpiredSignatureError:
        raise HTTPException(status_code=401, detail="Token 已过期")
    except jwt.InvalidTokenError:
        raise HTTPException(status_code=401, detail="Token 无效")
```

```python
def search_knowledge(db: Session, query: str, limit: int = 5) -> List[str]:
    # 分词
    keywords = [k for k in query.split() if len(k) >= 2]
    results = []
    seen_titles = set()
    
    for keyword in keywords:
        items = db.query(KnowledgeBase).filter(
            KnowledgeBase.is_active == 1,
            (KnowledgeBase.content.like(f"%{keyword}%")) |
            (KnowledgeBase.title.like(f"%{keyword}%")) |
            (KnowledgeBase.keywords.like(f"%{keyword}%"))
        ).order_by(KnowledgeBase.created_at.desc()).limit(limit).all()
        
        for item in items:
            if item.title not in seen_titles:
                results.append(item.content)
                seen_titles.add(item.title)
    
    return results[:limit]
```

```python
def _get_session_context(self, session_id: str, db: Session, max_turns: int = 3) -> str:
    # 查询最近 N 轮对话
    dialogues = db.query(Dialogue).filter(
        Dialogue.session_id == session_id
    ).order_by(Dialogue.created_at.desc()).limit(max_turns).all()
    
    if not dialogues:
        return ""
    
    # 反转顺序（从旧到新）
    dialogues = list(reversed(dialogues))
    
    # 格式化为对话历史
    context_parts = []
    for d in dialogues:
        context_parts.append(f"用户：{d.user_message}")
        context_parts.append(f"AI: {d.ai_response}")
    
    return "\n".join(context_parts)
```

### 5.5.5 回复后处理

```python
def _post_process_response(self, text: str) -> str:
    # 检测结束场景，强制返回简短回复
    if any(kw in text.lower() for kw in ['谢谢', '感谢']):
        if '请问' in text:
            idx = text.find('请问')
            text = text[:idx].strip() + '。祝您早日康复！'
    
    # 截断过长回复
    if len(text) > 500:
        for punct in ['。', '！', '？']:
            idx = text.find(punct, 300)
            if idx > 0:
                text = text[:idx+1]
                break
    
    # 移除追问
    if '请问' in text:
        idx = text.find('请问')
        text = text[:idx].strip()
    
    # 清理多余换行
    text = re.sub(r'\n{2,}', '\n', text)
    
    return text.strip()
```

```javascript
      url: `${baseUrl}/appointments/patient/${id}/status?status=confirmed`,
      method: 'PATCH',
      header: { Authorization: `Bearer ${token}` },
      success: () => {
        wx.showToast({ title: '确认成功' });
        this.loadAppointments();
      }
    });
  }
```

## 5.8 开发难点与解决方案

### 5.8.1 端口占用问题

**问题**：开发过程中经常遇到 8000 端口被占用的问题。

**解决方案**：开发 `stop_all.bat` 脚本强制停止所有 Python 进程：

```batch
@echo off
taskkill /F /IM python.exe
taskkill /F /IM streamlit.exe
echo 所有服务已停止
```

### 5.8.2 跨域问题

**问题**：小程序请求后端 API 时遇到跨域错误。

**解决方案**：在后端配置 CORS 中间件：

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],  # 生产环境应限制具体域名
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

### 5.8.3 Token 认证问题

**问题**：Token 过期后前端未正确处理，导致请求失败。

**解决方案**：前端封装统一的 API 客户端，自动处理 Token 过期：

```javascript
function request(options) {
  return wx.request({
    ...options,
    header: { Authorization: `Bearer ${getToken()}` },
    fail: (res) => {
      if (res.statusCode === 401) {
        // Token 过期，跳转登录
        wx.navigateTo({ url: '/pages/login/login' });
      }
    }
  });
}
```

### 5.8.4 数据加载不一致问题

**问题**：前端传 `page/page_size`，后端期望 `skip/limit`，导致数据加载失败。

**解决方案**：统一参数规范，API 客户端转换参数：

```python
@router.get("/")
async def get_dialogues(page: int = 1, page_size: int = 10):
    skip = (page - 1) * page_size
    limit = page_size
    dialogues = db.query(Dialogue).offset(skip).limit(limit).all()
    return dialogues
```

# 结论

## 工作总结

本文设计并实现了一套基于 AI 智能客服的牙科修复复诊提醒与管理系统。系统采用 FastAPI 后端框架、微信小程序和 Streamlit 医护后台的多端协同架构，核心功能包括 AI 智能对话、复诊计划管理、知识库浏览和数据统计。

主要工作包括：

1. **系统设计与实现**：完成了数据库设计、后端 API 开发、微信小程序开发和 Streamlit 医护后台开发，实现了 45+ 个 API 接口和 10 个前端页面。

2. **AI 模型微调**：使用 LoRA 微调技术对 Qwen2.5-7B-Instruct 模型进行牙科修复领域适配，使用 500 条专业问答数据进行训练，模型在测试集上准确率达到 90%。

3. **知识库构建**：从《口腔修复学》第 8 版整理了 804 条专业知识，按 7 个类别组织，支持关键词检索和分类浏览。

4. **系统测试**：完成了 82 个测试用例，包括功能测试、性能测试和安全测试，通过率达到 100%。

## 系统特色与创新点

1. **AI 智能客服**：基于 LoRA 微调的 Qwen2.5-7B 模型，结合 804 条专业知识库实现检索增强的回复生成，提供专业、准确的术后指导。

2. **多端协同**：微信小程序（患者端）、Streamlit Web（医护端）、FastAPI 后端（统一 API）三端协同，数据实时同步。

3. **安全机制**：JWT Token 认证、bcrypt 密码加密、患者数据隔离、管理员权限验证，确保系统安全性。

4. **人机协同**：AI 无法处理的问题支持人工接管，医护人员可以查看对话记录并进行人工回复。

## 不足之处

1. **AI 响应时间**：AI 对话接口平均响应时间 2.5 秒，主要原因是模型推理耗时较长，未来可考虑模型量化、蒸馏等优化技术。

2. **请求限流**：系统未实现 API 请求频率限制，存在被恶意调用的风险，未来可添加 Redis 限流中间件。

3. **知识库检索**：当前使用关键词匹配进行知识检索，准确性有限，未来可考虑使用稠密向量检索（如 DPR、Sentence-BERT）提升检索效果。

4. **自动备份**：数据库未配置自动备份，存在数据丢失风险，未来可配置定时备份脚本。

## 未来展望

1. **功能扩展**：增加智能分诊、用药提醒、满意度调查等功能，提升系统实用性。

2. **技术优化**：探索模型量化、知识蒸馏等技术，降低 AI 推理延迟；引入向量检索，提升知识检索准确性。

3. **推广应用**：与口腔医疗机构合作，进行实际部署和应用，收集用户反馈，持续优化系统。

4. **研究深入**：探索大语言模型在更多医疗场景的应用，如术前评估、治疗方案推荐、医患沟通辅助等。

```python
class AIService:
    def generate_response(self, user_message: str, patient_id: int, session_id: str, db: Session) -> str:
        # 1. 获取对话历史
        context = self._get_session_context(session_id, db, max_turns=3)
        
        # 2. 获取患者信息
        patient_info = self._get_patient_info(patient_id, db)
        
        # 3. 检索相关知识
        knowledge = self.search_knowledge(db, user_message, limit=3)
        
        # 4. 构建 System Prompt
        system_prompt = self._build_system_prompt(knowledge, patient_info)
        
        # 5. 调用 AI 服务
        return self._call_autodl_api(user_message, system_prompt, context)
```
